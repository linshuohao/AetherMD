# 文档维护规则

本页定义 AetherMD 文档如何演进。

## 归属边界

| 分区     | 负责                                              | 不负责                  |
| -------- | ------------------------------------------------- | ----------------------- |
| 架构文档 | 原则、边界、兼容策略、能力范围、CI 门禁           | 插件类型级契约          |
| SDK      | 插件可见的 Manifest、生命周期、命令、上下文、示例 | 运行时实现策略          |
| 工程文档 | 运行时行为、性能、安全、可观测性、并发            | 产品愿景或公开 API 变更 |
| ADR      | 决策历史与取舍                                    | 教程内容                |
| 社区文档 | 贡献模型与治理                                    | 技术契约                |

## 修改规则

1. 修改最小且最准确的归属分区。
2. 术语变化要同步更新 [核心词汇表](glossary.md)。
3. 持久取舍或推翻既有决策时，需要新增 ADR。
4. 谨慎使用 RFC 关键词：偏好不要写成 MUST。
5. SDK 契约变化要先更新 SDK 文档，再更新工程说明。
6. 如果实现策略与架构原则冲突，应更新或废弃相关 ADR。
7. 文档以当前能力事实为准，不维护过程性发布历史。

## 文档策略

当前 `docs/` 下的页面就是活跃维护面。不要维护多套事实来源。行文以最终目标与专业成果输出为核心，避免过程性、阶段性叙事。

## 变更前检查

- 每个公开契约都有唯一归属文档。
- 每个已接受的架构决策都出现在 [ADR](adr/README.md)（含 [ADR 009：发布与治理](adr/009-release-governance.md)）。
- [v1.0 能力范围](architecture/roadmap.md) 与 [实施范围](engineering/mvp-implementation-plan.md) 条目能映射到包或能力域。
- [版本兼容策略](architecture/compatibility.md) 记录 public export 变更。
- CI 校验计划足够具体，可自动化（见 [CI 校验计划](architecture/ci-checklist.md)）。
- 发布与许可证决策已决议或显式标注待落地项（见 [发布流程](community/release-process.md)）。
